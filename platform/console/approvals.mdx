# Approvals

Every action classified [tier T2 or higher](/protocol/reference/approval-tiers) routes to an Approval Tray. The Tray is a Console surface and an [Avatar](/protocol/layers/avatar) implementation; the same approval can render in [Face](/face/overview), Slack, email, or SMS.

## What an approver sees

For each pending approval:

- **Action** — what's about to happen (action class, parameters, target)
- **Justification** — the agent's rationale, captured by Bridge
- **Preview** — a render of the side effect (e.g. an email about to send shows the rendered email; a payment about to fire shows the payment summary; a CRM write shows the diff)
- **Tier** — risk classification
- **Token scope** — what the agent is currently authorized for
- **Expires at** — the approval window deadline

Three actions:

- **Approve** — allow as-proposed
- **Deny** — refuse with a reason
- **Modify** — narrow the action and approve (subject to attenuation; can shrink, never widen)

## Routing

Approvals can route to multiple transports simultaneously. The first response wins.

```yaml
- match: { tier: 4 }
  transports:
    - { kind: console, workspace: operations }
    - { kind: slack,   channel:   "#ops-approvals" }
    - { kind: sms,     phone:     "+1-555-..." }
```

Routing is configured per tenant in `routing.yaml`.

## Identity

Every approval response is attributed to an `approver_id` issued by the tenant's identity provider. Anonymous approvals are not supported.

## SLA

Tier-defined defaults (configurable per tenant):

| Tier | Default window | On timeout |
|---|---|---|
| T2 | 4 hours | Default-deny, ledger as `escalation_timeout` |
| T3 | 1 hour | Default-deny, escalation alert |
| T4 | 24 hours | Default-deny, escalation alert |

Median approval-to-action time across current Operators: **94 seconds**.

## Bulk actions

For high-volume scenarios (e.g. 200 templated outbound sends), the Console supports bulk approve/deny with mandatory inspection of a sampled subset. Bulk approvals attribute to a single approver_id; the underlying ledger entries record the bulk batch reference.

## Late responses

If approval arrives after the window, the action is already ledgered as denied. The late response is recorded as `late_response` and ignored for action purposes. Late responses still show in the approver's history.
