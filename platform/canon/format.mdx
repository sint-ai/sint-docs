# Canon format

A Canon repo is a directory tree with three required artifacts:

1. **`CANON.yaml`** — manifest declaring version, scopes, read conventions
2. **Content files** — markdown (`.md`), YAML (`.yaml`), JSON (`.json`)
3. **An indexed manifest** — produced by the Canon indexer, content-addressed, used for hash-chain inclusion in the [Evidence Ledger](/protocol/reference/evidence-ledger)

## Content files

### Markdown

The default format. Standard CommonMark plus frontmatter:

```markdown
---
canon_id: marketing.positioning-brief
canon_version: 3
last_updated: 2026-04-01
maintainer: marketing-lead
read_audit: false
---

# Positioning brief

We sell to US mid-market services firms ($3M–$20M revenue, 15–80 employees) ...
```

Frontmatter fields:

- `canon_id` — stable identifier; never change after first publish
- `canon_version` — bumped by the maintainer on substantive change (per-document semver)
- `last_updated` — auto-derived from Git
- `maintainer` — handle/role responsible
- `read_audit` — true if reads should be ledgered (default per scope)

### YAML

Used for structured data (rubrics, type definitions, jurisdictional rules):

```yaml
# legal/jurisdictional-flags.yaml
canon_id: legal.jurisdictional-flags
canon_version: 7
maintainer: legal-counsel

flags:
  - jurisdiction: NY
    reason: "Employment law specifics"
    requires_review_by: duty-attorney
  - jurisdiction: CA
    reason: "Privacy law specifics (CCPA, CPRA)"
    requires_review_by: privacy-officer
```

### JSON

Used for schemas:

```json
{
  "canon_id": "schemas.contract-summary",
  "canon_version": 4,
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "parties": { "type": "array", "items": { "type": "string" } },
    "effective_date": { "type": "string", "format": "date" }
  }
}
```

## Path conventions

Path resolution is **strict prefix matching**. `canon.read({ path: "sales/scoring-rubric.md" })` resolves to exactly that file. There is no fuzzy matching.

Wildcards (`marketing/*.md`, `sales/objection-handling/**/*.md`) are supported but the calling Pipeline must declare them in its `imports`.

## Versioning model

Two layers of versioning:

1. **Per-document version** — the `canon_version` field. Maintainers bump on substantive change. Pipelines can pin: `canon.read({ path: "...", version: 7 })`.
2. **Repo-level version** — the Git commit hash. Indexer produces a content-addressed manifest per commit.

A Pipeline that needs reproducibility against a specific Canon state pins to a commit hash. A Pipeline that wants the latest pins to a branch (typically `main`).

## Reading from Canon

```typescript
import { canon } from "@sint/canon";

// Latest version of the document
const brief = await canon.read({ path: "marketing/positioning-brief.md" });

// Specific document version
const oldBrief = await canon.read({ path: "marketing/positioning-brief.md", version: 5 });

// Specific Canon commit (full reproducibility)
const pinnedBrief = await canon.read({
  path: "marketing/positioning-brief.md",
  commit: "a3f2..."
});

// YAML or JSON
const flags = await canon.read({ path: "legal/jurisdictional-flags.yaml" });
```

All `canon.read` operations include a content hash in the result; that hash is what the [Evidence Ledger](/protocol/reference/evidence-ledger) records when the read is ledgered.
