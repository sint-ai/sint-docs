# Cron

Cron is the trigger system. Two flavors: scheduled (cron syntax) and event-driven (subscribe to a queue, webhook, or system signal).

## Scheduled triggers

Standard cron syntax with seconds precision. Per-tenant timezone.

```yaml
# pipelines/cmo-weekly.yaml
triggers:
  - kind: schedule
    cron: "0 6 * * MON"          # every Monday at 06:00 tenant-local
    pipeline: cmo-week-plan
```

Triggers are themselves capability-token-gated. A schedule triggering a T1 Pipeline needs a token with `tier_max ≥ 1`; T4 schedules require an explicit human-issued long-lived token (rare).

## Event triggers

Subscribe to webhooks, queues, or internal Platform events.

```yaml
triggers:
  - kind: webhook
    path: "/webhooks/docusign-signed"
    auth: { kind: hmac, secret_ref: "secrets/docusign" }
    pipeline: onboarding

  - kind: event
    source: "engine.outcome"
    filter:
      action_class: "doc.summary.write"
      tier: 2
    pipeline: post-summary-followup
```

The `engine.outcome` source emits Platform events for every ledgered Outcome; Pipelines can chain on these without polling.

## Observability

Console surfaces every trigger fire under `Platform → Cron → Activity`. Each fire produces a ledger entry (`trigger.fired`) with the trigger config hash and the resulting Run trace_id.

## Failure modes

| Failure | Behavior |
|---|---|
| Pipeline absent or invalid | Trigger fires; first stage logs `pipeline_invalid`; alert raised |
| Token expired | Trigger fires; Bridge issues a new token via the trigger's renewal config; if renewal fails, alert |
| Webhook signature invalid | Trigger refused; ledgered as `webhook_signature_invalid` |
| Schedule overlap (previous run still active) | Configurable: `skip` (default), `queue`, or `parallel` |

## Drift detection

Cron compares each scheduled trigger's actual fire times against the schedule's expected fire times. Drift > 30 seconds at p99 over a rolling 24h window raises an alert.
