# Skills

A Skill is an LLM-callable function plus everything it needs to be safe to call: a prompt, tests, capability requirements, tier classification, and an output schema.

Skills are smaller than [Components](/platform/commons/components). A Component is a Pipeline sub-graph. A Skill is a single agent capability the Component or Pipeline can invoke.

## Anatomy

```
@commons/extract-contract-fields/
  manifest.yaml
  prompt.md              the prompt template
  schema.json            output schema (JSON Schema)
  tests/                 input/output pairs + assertions
  README.md
```

Example manifest:

```yaml
name: extract-contract-fields
version: 1.4.0
description: |
  Extracts party names, effective date, term, governing jurisdiction,
  and key obligations from a contract document.
inputs:
  - { name: text, type: string }
outputs:
  - { name: fields, type: object, schema: "schema.json" }
capabilities:
  - "doc.read"
tier: 1
confidence:
  floor: 0.78
  bump_below_to_tier: 2
llm:
  preferred_model: "claude-sonnet-4-6"
  fallback_model: "gpt-5"
  max_tokens: 1500
  temperature: 0
tests:
  required_pass_rate: 0.95
```

## Why skills

Three reasons:

1. **Prompts are versioned.** A change to the extraction prompt is a semver bump, with tests, with a changelog. No "we tweaked it yesterday and revenue dropped."
2. **Capabilities are explicit.** A Skill declares what scopes it needs. The capability token must include them; the Gate enforces.
3. **Tier impact is declared.** A Skill can declare that low-confidence outputs bump the tier, ensuring escalation when the model is uncertain.

## Provider abstraction

Skills declare a `preferred_model` and a `fallback_model`. The Skill runtime resolves both at execute time; if the preferred is unavailable or budget-exceeded, the fallback is used. ROSClaw's findings argue against assuming model fungibility, so the Skill's tests are run against both at version-tag time.

## Standard library

Selected skills shipped in `@commons/*`:

| Skill | Purpose |
|---|---|
| `@commons/extract-contract-fields` | Contract intake |
| `@commons/classify-document` | Generic doc classification |
| `@commons/draft-outbound-email` | Outbound email draft from a structured prompt |
| `@commons/parse-reply-intent` | Inbound reply intent classification |
| `@commons/summarize-call` | Voice call summary |
| `@commons/generate-welcome-packet` | Onboarding doc generation from Canon templates |
| `@commons/draft-social-post` | Social post draft from a positioning brief |
| `@commons/rate-content-against-brief` | Content QA against a Canon brief |

## Authoring

Open a PR against the tenant's Commons repo (or against [`github.com/sint-ai/commons`](https://github.com/sint-ai/commons) for upstream). CI runs the Skill's test suite against the preferred and fallback models. Pass rate must meet `tests.required_pass_rate` to merge.
