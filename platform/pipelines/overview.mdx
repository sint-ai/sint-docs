# Pipelines

A Pipeline is a YAML declaration of an Operator's behavior. The Pipeline compiler turns the YAML into an executable graph; the Engine runs the graph; the Gate enforces; the Ledger records.

Pipelines are how Operators are built on the Platform. Every Operator we ship is a Pipeline (often composed of [Components](/platform/commons/components) and [Skills](/platform/commons/skills) from [Commons](/platform/commons/overview)).

## A minimal Pipeline

```yaml
# pipelines/hello-world.yaml
name: hello-world
version: 0.1.0
operator: HelloWorld

triggers:
  - kind: webhook
    path: "/webhooks/hello"

stages:
  - id: greet
    skill: "@commons/echo@^1.0.0"
    inputs: { message: ${trigger.payload.message} }

  - id: report
    action:
      class: "report.write"
      params: { text: ${greet.output} }
    tier: 1
```

Two stages, one trigger. The Pipeline compiler validates this against the schema and against [M8 Guardrails](/platform/overview#m8-guardrails) before deploy.

## What a Pipeline declares

- **Name + version** — semver per Pipeline
- **Operator** — the Operator class this Pipeline implements (drives metrics naming)
- **Triggers** — when to run (see [Cron](/platform/cron/overview))
- **Stages** — the executable graph
- **Imports** — pinned versions of Commons artifacts
- **Action classes** — what action classes this Pipeline issues (Gate enforces; deviations rejected)
- **Capabilities** — what capability scopes this Pipeline's tokens need
- **Outputs** — what systems of record it writes to

## Compilation

`@sint/pipelines/compile` turns the YAML into a `PipelineGraph`:

- Validates the YAML against the schema
- Resolves all `imports` and pins versions
- Type-checks `${...}` references across stages
- Computes the action class set and verifies against the Pipeline declaration
- Computes the capability set and verifies the Pipeline's token request matches
- Runs M8 Guardrails

[Compilation details →](/platform/pipelines/compilation)

## Execution

The compiled graph is executed by the Engine. Each stage:

- Resolves its inputs
- If it's a `skill` or `component` invocation, dispatches to the runtime
- If it's an `action`, submits to the [Gate](/protocol/layers/gate)
- On `allow`, the Engine commits to the [Ledger](/protocol/reference/evidence-ledger), then dispatches to the [Bridge](/protocol/layers/bridge) adapter
- On `escalate`, the stage waits for approval (subject to its `timeout`)
- On `deny`, the Pipeline's failure path activates (configurable per stage)

## Hot reload

Pipeline updates deploy via the Console (or `sint deploy <pipeline>`). New runs use the new version; in-flight runs complete on the version they started with.

A Pipeline deploy is itself ledgered (`pipeline.deployed`).
