# Memory

Memory is the per-tenant working state for Operators. Where Canon is read-only knowledge maintained by humans, Memory is read/write state maintained by Operator runs.

## What it stores

- Per-Operator working state (cursors, current run state, partial results)
- Per-tenant counters and aggregates not warranting a full database (rolling totals, deduplication keys)
- Per-trace ephemeral state across stages of a Pipeline
- Last-known-good outputs for fallback during partial outages

## What it does not store

- The Evidence Ledger — that's [Engine](/protocol/layers/engine) territory and is immutable
- Long-term knowledge — that's [Canon](/platform/canon/overview)
- Tenant data of record — that's the client's CRM, document store, etc.
- Capability tokens — those live in the [token storage](/protocol/reference/capability-tokens)

## Versioning

Every write produces a new version with the previous version's hash. Memory is content-addressable; a read can target the latest or a specific version. Rollback is a single API call.

## Snapshots

Snapshots are taken automatically every 15 minutes per tenant and on manual request. Snapshots are exportable to client-controlled storage on the same schedule as the Evidence Ledger.

## Access

```typescript
import { memory } from "@sint/memory";

await memory.set({ tenant: "acme", key: "revenue-op:cursor", value: { last_id: "abc" } });
const v = await memory.get({ tenant: "acme", key: "revenue-op:cursor" });
const history = await memory.history({ tenant: "acme", key: "revenue-op:cursor", limit: 10 });
await memory.rollback({ tenant: "acme", key: "revenue-op:cursor", to_version: history[3].version });
```

Memory access is itself ledgered for `tier ≥ 2` keys (declared per-key). Read access at T0 is not ledgered by default to avoid log explosion; this is configurable.

## Storage

Per-tenant Postgres (default). Hot keys cached in Redis with cache invalidation on every write. Capacity defaults: 5GB per tenant; expandable.

## Memory and Canon

Memory and [Canon](/platform/canon/overview) are intentionally separate.

| Memory | Canon |
|---|---|
| Read/write | Read-only at runtime |
| Maintained by Operators | Maintained by humans |
| Versioned by writes | Versioned by Git commits |
| Ephemeral / working | Long-lived knowledge |
| Per-Operator scoped (mostly) | Tenant-scoped, shared across Operators |

A Pipeline that wants to remember "what we last sent this prospect" uses Memory. A Pipeline that wants to read "our positioning brief" uses Canon.
