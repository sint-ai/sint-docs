# Memory + Canon integration

Memory and [Canon](/platform/canon/overview) serve different purposes (working state vs. knowledge), but Operators frequently need both in the same step. The Platform provides three integration patterns.

## Pattern 1: Canon snapshots in Memory

For Pipelines that need to read the same Canon document many times in one Run, the Canon Component can snapshot to Memory at the start of the Run. Subsequent reads in the same Run hit Memory (microsecond latency) instead of going back to the Canon repo (ms latency).

```yaml
stages:
  - id: brief
    component: "@commons/canon-fetch@^1.0.0"
    inputs:
      path: "marketing/positioning-brief.md"
      snapshot_to_memory: true       # snapshot for the duration of this Run

  - id: draft
    skill: "@commons/draft-social-post@^1.2.0"
    inputs:
      brief: ${brief.snapshot}        # reads from Memory
```

The snapshot is content-addressed; if the underlying Canon document hasn't changed since the last snapshot, the existing snapshot is reused.

## Pattern 2: Memory join with Canon

When Memory holds operational state and Canon holds the rules that govern that state, the Platform's `join` helper lets agents read both atomically.

```typescript
const view = await join({
  tenant: "acme",
  memory_keys: ["revenue-op:cursor", "revenue-op:rate-counters"],
  canon_paths: ["sales/policies.md", "sales/scoring-rubric.md"],
});
// view.memory.{key} and view.canon.{path} populated atomically
```

Atomic in the read sense: both are pinned to a single timestamp and a single Canon revision.

## Pattern 3: Memory as Canon staging

For Pipelines that propose updates to Canon (e.g. "this contract clause looks like a new pattern; should we add it to the playbook?"), Memory is the staging ground.

The Pipeline writes the proposal to Memory, opens an [approval](/platform/console/approvals), and on approve, an [Engineer](/platform/console/overview#roles) merges a PR that lifts the proposal into Canon.

```yaml
stages:
  - id: pattern-detect
    skill: "@commons/detect-contract-pattern@^1.0.0"
    inputs: { contract: ${intake.payload} }

  - id: stage-proposal
    when: ${pattern-detect.is_new_pattern}
    action:
      class: "memory.write"
      params:
        key: "canon-proposals/contracts/${pattern-detect.id}.md"
        value: ${pattern-detect.proposal}
    tier: 2
    approval:
      route: { kind: console, workspace: legal }
```

Approved proposals show in the Console under `Canon → Pending`. An Engineer reviews, edits, opens a Canon PR, and merges. Canon is now updated; the next Run reads the new pattern.
