# Architecture

This page covers the data path through a single Protocol-governed request and the boundaries that make the safety guarantees hold.

## Single-request data path

```
Agent intent
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│  L1 — Bridge                                             │
│  • Translate agent intent → Protocol-shaped action       │
│  • Issue/inspect capability token                        │
│  • Result<Action, ErrBridge>                             │
└──────────────────────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│  L2 — Gate (Policy Gateway)                              │
│  • Token attenuation check (I-T1)                        │
│  • Policy evaluation                                     │
│  • Tier classification (T0–T4)                           │
│  • If T2+ → enqueue for human approval                   │
│  • Decision: allow | deny | escalate                     │
│  • Pre-write decision to Engine ledger queue (I-G3)      │
│  • Result<Decision, ErrGate>                             │
└──────────────────────────────────────────────────────────┘
   │
   ▼ (only if allowed)
┌──────────────────────────────────────────────────────────┐
│  L3 — Engine                                             │
│  • Commit decision to Evidence Ledger (INSERT-only)      │
│  • Execute the Action against Bridge adapter             │
│  • Append outcome to ledger                              │
│  • Optional: emit ProofReceipt to attestation hook       │
│  • Result<Outcome, ErrEngine>                            │
└──────────────────────────────────────────────────────────┘
   │
   ▼ (optional)
┌──────────────────────────────────────────────────────────┐
│  L4 — Economy (optional)                                 │
│  • Meter usage                                           │
│  • Settle if a settlement rail is configured             │
│  • Failure here does NOT block the request (I-F1)        │
└──────────────────────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│  L5 — Avatar (optional surface)                          │
│  • Surface decision/outcome to human operator            │
│  • Render approval tray for queued T2+ items             │
└──────────────────────────────────────────────────────────┘
```

## Trust boundaries

```
                 ┌────────────────────────────────┐
   UNTRUSTED →   │  Agent (LLM, planner, model)   │
                 └─────────────────┬──────────────┘
                                   │
                 ╔═════════════════▼══════════════════╗
   SAFETY CORE   ║  Bridge → Gate → Engine            ║
                 ║  (Result-typed, append-only,       ║
                 ║   no exceptions cross the boundary)║
                 ╚═════════════════┬══════════════════╝
                                   │
                 ┌─────────────────▼──────────────────┐
   ACTUATOR →    │  Robot · Money · Tool · Inbox      │
                 └────────────────────────────────────┘
```

The agent is **always treated as untrusted**, regardless of which model is running. ROSClaw's empirical finding — up to 4.8× spread in out-of-policy proposals across frontier backends under identical safety envelopes — is the empirical justification for this stance.

## Package boundaries

| Package | Purpose | Consumes | Produces |
|---|---|---|---|
| `@sint/tokens` | Capability token primitives, attenuation logic, revocation index | — | Tokens, attenuation results |
| `@sint/bridge` | Adapters: agent ↔ actuator translation per integration | `@sint/tokens` | `Action` objects |
| `@sint/gateway-server` | HTTP service; runs the Gate | `@sint/tokens`, `@sint/bridge`, `@sint/engine` | `Decision` records |
| `@sint/engine` | Execution runtime + Evidence Ledger | `@sint/tokens`, `@sint/bridge` | Outcomes, ledger entries |
| `@sint/economy-api` | Optional metering/settlement interface | `@sint/engine` (events) | Usage records |
| `@sint/avatar` | HITL surface protocol | `@sint/gateway-server` (approvals) | Approvals, denials |
| `@sint/policy-dsl` | Policy language compiler/runtime | — | Policies |

Storage adapters (`@sint/storage-postgres`, `@sint/storage-redis`, `@sint/storage-memory`) implement a single interface (I-P1) consumed by Engine and Gate.

## Concurrency model

- **Gate** is stateless per-request; horizontally scalable behind any L7 load balancer.
- **Engine** owns the ledger write order. A single ledger instance per tenant; partition by tenant for horizontal scale across tenants.
- **Token revocation** propagates via Redis pub/sub with a Postgres consistency fallback. Measured <50ms p99 propagation across three Railway regions on production workloads.
- **Approval queue** is a separate event stream; approvals attach to the ledger entry by ID, not by overwriting.

## Failure modes

| Failure | Behavior | Invariant |
|---|---|---|
| Agent proposes out-of-scope action | Gate denies; ledgered as `denied` | I-T1 |
| Gate cannot reach Postgres | Fail-closed: deny all writes; reads continue | I-F1 (safety-critical path) |
| Engine cannot reach Economy meter | Fail-open: action proceeds; metering failure logged separately | I-F1 (non-safety path) |
| Approval timeout (no human response within tier-defined window) | Default-deny; ledgered as `escalation_timeout` | I-G3 |
| E-stop signal received | All non-terminal actions halt; tokens invalidated tenant-wide | I-G2 |
| Ledger hash chain divergence detected | Engine refuses new writes; alerts pager; quarantine procedure activates | I-G3 |

## Performance characteristics

Measured on `@sint/gateway-server` v0.8 against synthetic load simulating a 200-agent fleet (3 Railway regions, single Postgres primary per tenant):

| Metric | Value |
|---|---|
| Throughput (per Gateway instance) | 7,400 req/s |
| Latency p50 | 4 ms |
| Latency p99 | 12 ms |
| Token revocation propagation p99 | 47 ms (3-region) |
| Ledger write durability | acknowledge-after-pending-commit |

Benchmark harness: [`benchmarks/gateway-load`](https://github.com/sint-ai/sint-protocol/tree/main/benchmarks/gateway-load).

## Architectural decisions

The Protocol's architecture decisions are recorded in the repo at [`docs/adr/`](https://github.com/sint-ai/sint-protocol/tree/main/docs/adr). Each ADR captures the context, the decision, and the consequences. Notable:

- ADR-0002 — The five invariants are non-negotiable
- ADR-0003 — The ledger is append-only and SHA-256 hash-chained
- ADR-0004 — Tokens encode physical constraints, not external config
- ADR-0005 — The Economy layer is optional
- ADR-0006 — `Result` types, not exceptions, across the safety boundary
- ADR-0007 — Property tests are required for invariant code
