# Economy (Layer 4)

Economy is the optional metering and settlement layer. **The Protocol is fully usable without it.**

<Note>
**SINT Labs does not issue, sell, or plan to issue a token.** Economy is an interface for optional metering and settlement; clients choose their own settlement rail (Stripe, ACH, on-chain, internal accounting) or omit the layer entirely. Most production deployments today omit Layer 4. See [/origin](/origin).
</Note>

## When you'd want Economy

Three concrete scenarios:

1. **Per-action billing.** An Operator sells access to a workflow; the Operator's owner wants to meter usage for invoicing.
2. **Cross-entity settlement.** A machine or agent pays for a service rendered by another machine or agent. Common in robotics-as-a-service, machine-economy R&D, or multi-tenant API metering.
3. **Spend governance.** A capability token has a `spend_cap_usd`; Economy is the meter that decrements it as actions accrue cost.

## When you'd skip Economy

- You bill per-seat, per-Operator, or by retainer (most B2B Operator deployments).
- You handle metering in your own system of record (your ledger, your data warehouse).
- You don't need cross-entity settlement.
- You're integrating only the safety layers (1–3) for an internal use case.

You can adopt Economy later without reshaping anything in Layers 1–3.

## What Economy does

- Receives `Outcome` events from Engine (fire-and-forget; failure here never blocks the safety path — [I-F1](/protocol/invariants/i-f1-fail-modes))
- Maps each Outcome to a `UsageRecord` per the active `MeterPlan`
- Emits `UsageRecord` events to a configurable settlement rail
- Maintains rolling per-tenant spend totals for `spend_cap_usd` enforcement (read by Gate)

## Interfaces

```typescript
type UsageRecord = {
  tenant: string;
  agent_id: string;
  action_class: string;
  ledger_id: string;             // links back to Engine's Evidence Ledger
  units: number;                 // domain-specific: requests, tokens, bytes, etc.
  unit_price_usd: number;
  total_usd: number;
  ts: string;
};

interface MeterPlan {
  rate(action: Action, outcome: Outcome): UsageRecord | null;
}

interface SettlementRail {
  emit(record: UsageRecord): Promise<Result<SettlementReceipt, SettlementErr>>;
}
```

## Reference rails

The Protocol ships a no-op rail (`@sint/economy-noop`, useful for testing) and the API surface — but **does not bundle a settlement rail.** Bring your own.

Common patterns:

- **Stripe metered billing.** Implement `SettlementRail` to call `stripe.subscriptionItems.createUsageRecord()` per record. Cost: zero on-chain footprint.
- **Internal data warehouse.** Implement `SettlementRail` to write to your warehouse (Snowflake, BigQuery, Postgres). Cost: $0 marginal.
- **On-chain settlement.** If you need decentralized machine-economy settlement (rare), implement `SettlementRail` against the chain of your choice. The Protocol is rail-agnostic and explicitly does not endorse any specific chain.
- **Hybrid.** Most production deployments emit to two rails — internal warehouse + a customer-visible billing system — for reconciliation.

## Failure semantics

Per [I-F1](/protocol/invariants/i-f1-fail-modes):

- A settlement rail failure **never blocks** the safety path. Engine completes the action; the failed UsageRecord is queued for retry.
- A meter plan misconfiguration produces a `null` record (the action is not metered) and an alert.
- `spend_cap_usd` enforcement on Gate reads the **last successful** rolling spend total. If Economy is unreachable, Gate falls back to the most recent cached value with a configurable staleness threshold; past the threshold, Gate fails closed on spend-cap-bound actions.

## Spend cap example

A capability token can carry `spend_cap_usd: 50`. Economy maintains the rolling total per token. Gate reads the total before allowing an action whose meter plan would push the total past the cap.

```yaml
# policies/spend-bound.yaml
- match:
    action_class: "llm.complete"
  effect: allow
  tier: 1
  spend_check: token.spend_cap_usd
```

When the cap is hit, subsequent allow-eligible requests return `deny` with `reason: "spend_cap_exceeded"`. The action itself never fires.

## On-chain settlement (if you really need it)

If your use case requires decentralized settlement — for example, multi-party machine economies where no central operator can clear — you can implement `SettlementRail` against the chain of your choice.

We do not bundle, endorse, or operate any specific chain. The Protocol's stance: chains are an **implementation detail of the rail**, not a feature of the Protocol.

The earlier SINT project (pre-pivot) integrated Solana, Ethereum, and Cosmos. None of that is part of the current Protocol. See [/origin](/origin).

## What Economy does not do

- Mint, sell, or distribute tokens.
- Define value or pricing — that's the meter plan, which you write.
- Block actions on settlement failure (per [I-F1](/protocol/invariants/i-f1-fail-modes)).
- Modify ledger entries — Economy reads outcomes, never writes them back to the safety ledger.
