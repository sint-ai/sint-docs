# Evidence Ledger

The Evidence Ledger is the audit and reconstruction substrate of the Protocol. Every Decision, every Outcome, every Approval, every Denial — every event the Protocol produces — is appended to a per-tenant ledger before any side effect can fire.

## Properties

- **Append-only.** Enforced at the database level via INSERT-only triggers. UPDATE and DELETE are rejected.
- **Hash-chained.** Each entry's hash is a SHA-256 over `prev_hash || canonical_json(payload)`. Tampering with any historical entry breaks the chain at every subsequent entry.
- **Continuously verified.** A verification job recomputes the chain every 10 minutes, comparing against an independently-maintained Redis index.
- **Exportable.** Per-tenant export to client storage on a configurable schedule. Clients can re-verify against their own export.
- **Tamper-evident.** Direct DB modification attempts are themselves ledgered as `tamper_attempt` entries.

## Entry shape

```typescript
type LedgerEntry = {
  id: string;                    // 0x-prefixed UUID
  tenant: string;
  trace_id: string;              // groups related entries (one per request)
  prev_hash: string;             // SHA-256 of previous entry
  hash: string;                  // SHA-256 of (prev_hash || canonical_json(this entry sans hash))
  ts: string;                    // ISO 8601, millisecond precision
  kind: LedgerKind;
  tier: 0 | 1 | 2 | 3 | 4;
  agent_id: string;
  action_class: string;
  payload: Record<string, unknown>;
  proof_receipt_id?: string;
};

type LedgerKind =
  | "decision_allow"
  | "decision_deny"
  | "decision_escalate"
  | "outcome_success"
  | "outcome_failure"
  | "outcome_timeout"
  | "approval_granted"
  | "approval_denied"
  | "approval_modified"
  | "estop_fired"
  | "estop_cleared"
  | "tamper_attempt"
  | "infra_failure"
  | "policy_loaded"
  | "token_issued"
  | "token_revoked";
```

## Hash chain

```
entry[0]:
  prev_hash = "0x0000...0000"  // genesis
  payload   = { ... }
  hash      = SHA256(prev_hash || canonical_json(payload))

entry[1]:
  prev_hash = entry[0].hash
  payload   = { ... }
  hash      = SHA256(prev_hash || canonical_json(payload))

entry[N]:
  prev_hash = entry[N-1].hash
  ...
```

Canonical JSON: keys sorted lexicographically, no whitespace, UTF-8, no trailing newlines. Implementation: `@sint/engine/canonical.ts`.

## Verification

```bash
pnpm --filter @sint/engine ledger:verify --tenant <id>
# Recomputes the chain against the active storage adapter and the Redis index.
# Exits 0 on success; non-zero on divergence.
```

For continuous verification in production:

```bash
pnpm --filter @sint/engine ledger:verify-continuous --tenant <id> --interval 10m
```

The continuous job emits Prometheus metrics:

- `sint_ledger_entries_total{tenant}`
- `sint_ledger_chain_intact{tenant}` (1 / 0)
- `sint_ledger_last_verify_ts{tenant}`

## Export

Per-tenant export to client-controlled storage on a schedule.

```yaml
# tenant config
export:
  destination:
    kind: s3
    bucket: client-audit
    prefix: sint-ledger/
  schedule: "0 0 * * *"   # daily at 00:00 UTC
  format: jsonl
  include_payloads: true
```

Supported destinations: S3, GCS, SFTP. Custom destinations implement a `LedgerExportSink` interface.

The export is itself ledgered: an `export_completed` entry records the destination, the entry range covered, and the final chain hash.

## Proof receipts

For tier-gated critical actions, the Engine can emit a `ProofReceipt` to a pluggable attestation hook. The receipt attaches verifiable-compute evidence (TEE attestation, signed enclave statement, ZK proof, etc.) to the ledger entry.

The Protocol does not prescribe an attestation backend.

```typescript
interface ProofReceiptHook {
  generate(entry: LedgerEntry): Promise<Result<ProofReceipt, ProofErr>>;
  verify(receipt: ProofReceipt, entry: LedgerEntry): Promise<Result<true, ProofErr>>;
}
```

Reference hooks:

- `@sint/proofs-noop` — signed timestamp; useful for development
- `@sint/proofs-aws-nitro` — AWS Nitro Enclave attestation
- `@sint/proofs-tdx` — Intel TDX attestation
- (Bring your own.)

ProofReceipt failures never block the safety path per [I-F1](/protocol/invariants/i-f1-fail-modes); the entry is committed without a receipt and an alert is raised.

## Reading the ledger

```bash
# REST
curl -sS https://gate.example.com/v1/ledger?tenant=acme&limit=100

# Filtered by trace
curl -sS https://gate.example.com/v1/ledger?tenant=acme&trace_id=abc-123

# Filtered by kind
curl -sS https://gate.example.com/v1/ledger?tenant=acme&kind=decision_deny&since=2026-04-01T00:00:00Z
```

Read access is governed by tenant-side RBAC. The default Console role grants read access scoped to the user's tenant; auditor roles can be granted read-only access to the full ledger.

## Schema migrations

The ledger schema is versioned. Schema migrations append a new column or table; they never modify or drop existing data. Old entries remain readable by older client versions.

## What the ledger does not contain

- **Agent reasoning traces.** The Protocol records what the agent decided, not how. If you want reasoning traces, capture them in the agent runtime and store them in your own observability system.
- **Full request/response bodies for every action.** The payload is canonicalized and may exclude large blobs (configurable per tenant). Blob references (S3 keys, content hashes) are included; blobs themselves can be exported separately.
- **PII beyond what the action requires.** Per tenant configuration, PII can be redacted before ledger commit.

## Production scale

In the [`@sint-protocol`](https://github.com/sint-ai/sint-protocol) workspace's production deployments, the ledger has crossed **1.24M+ entries** with **zero hash-chain drift** over the most recent 90-day window across all tenants.
