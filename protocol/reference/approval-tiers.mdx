# Approval tiers (T0–T4)

Every action is classified into one of five tiers. T0 and T1 are autonomous. T2+ require human approval. T4 is always human, never autonomous.

## The tiers

| Tier | Definition | Examples | Default routing |
|---|---|---|---|
| **T0** | Read-only, no state change | List records, fetch a document, query a calendar | Auto-allow |
| **T1** | Scheduled write within pre-approved scope | Send a templated email, file a ticket from a template, schedule a meeting in published availability | Auto-allow |
| **T2** | Ad-hoc write, or write under a client-defined "review" condition | Send a non-templated email, write to a CRM record, generate a custom document | Approval tray |
| **T3** | Multi-entity / cross-tenant / multi-recipient action | Send to >5 recipients, write to multiple downstream systems, cross-account operation | Approval tray + double-confirmation |
| **T4** | Irreversible or regulated | Financial settlement, physical motion above safety threshold, legal filings, deletes | Always human, never autonomous |

## Classification

Tier is determined by the policy bundle's `tier` declaration on each `match` rule. Default policies that ship with the Protocol classify common action classes; custom policies override or extend.

```yaml
# Examples
- match: { action_class: "doc.read" }
  tier: 0

- match: { action_class: "email.send", params: { template: { eq: "approved" } } }
  tier: 1

- match: { action_class: "email.send" }
  tier: 2

- match: { action_class: "email.send", params: { recipients: { count_gt: 5 } } }
  tier: 3

- match: { action_class: "payment.send" }
  tier: 4
```

Rules are evaluated top-to-bottom; the first match wins. The reference policy bundle defaults to the most restrictive tier when no rule matches (`tier: 4`, fail-closed).

## Approval routing

When a Decision is `escalate`, the Gate emits an `Approval` event. The event routes through one or more transports:

```yaml
- match: { tier: 4 }
  transports:
    - { kind: console,  workspace: operations }
    - { kind: slack,    channel:   "#ops-approvals" }
    - { kind: sms,      phone:     "+1-555-..." }

- match: { tier: 2, action_class: "doc.summary.write" }
  transports:
    - { kind: console, workspace: intake }
```

The first response wins. Late responses are ledgered as `late_response` and ignored.

## Approval timing

| Tier | Default window | On timeout |
|---|---|---|
| T2 | 4 hours | Default-deny, ledger as `escalation_timeout` |
| T3 | 1 hour | Default-deny, escalation alert |
| T4 | 24 hours | Default-deny, escalation alert |

Windows are configurable per-tenant and can be overridden per action class.

Median approval-to-action time across current Operator deployments: **94 seconds**.

## Approver identity

Approvers are identified by `approver_id`, issued by the tenant's identity provider (we recommend SSO/OIDC). The `approver_id` is recorded on the approval entry alongside the response. Anonymous approvals are not supported.

## Approval responses

```typescript
type ApprovalResponse =
  | { decision: "approve"; comment?: string; approver_id: string }
  | { decision: "deny";    reason: string;  approver_id: string }
  | { decision: "modify";  changes: ActionDiff; approver_id: string };
```

`modify` is supported only for action classes whose policy declares them modifiable. Modifications go through the same attenuation check as token delegation — the approver can narrow the action, never widen it.

## Tier overrides via token

A token's `tier_max` caps the maximum tier the token can authorize. A token with `tier_max: 1` denies all T2+ actions regardless of policy, scope, or approval availability. This is the "narrow agent" pattern: even if policy would allow T3 for this action class, this specific agent isn't trusted with that authority.
